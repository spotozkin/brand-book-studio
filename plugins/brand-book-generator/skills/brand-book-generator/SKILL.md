# Brand Book Generator

## What this skill produces

A single PDF brand book that a real person — the founder, the practice owner, their designer, their social media manager — could open and immediately use to make consistent decisions. The bar is not "looks impressive," it's "someone could hand this to a freelancer tomorrow and get on-brand work back."

A brand book from this skill always covers strategy, voice, visual system, applications, and governance at the depth its posture requires — but the document's SHAPE (which sections lead, what order, what register, how deep) is chosen per book in Steps 2.5–2.6, never inherited. Strategy always comes before aesthetics in the WORK, even when the finished document opens with color or voice. A beautiful palette attached to a fuzzy positioning is a worse outcome than a plain palette attached to a sharp one — if the brief is thin on strategy, push for clarity there before designing anything.

## Before you start: what this skill is and isn't

This skill produces a real, rendered visual system, not just documentation about one — but there's a genuine boundary. You can build an actual small geometric mark (see `references/mark_techniques.md`: a monogram badge, or a simple motif built from basic shapes like the candle/flame or mountain examples there), and you can build the full system around it: palette, type, UI components, a mini website mockup. What you can't do is generate finished illustrative or photographic artwork — a detailed character mascot, a painterly icon, a photoreal product shot. Be upfront about this distinction with the person: their brand book will contain a real, usable mark and a fully rendered system, but if they picture something illustrative rather than geometric, that's typically still a follow-up step (a designer, or an AI image tool) rather than something this skill produces directly.

**If the person has an existing logo, use the actual file — don't redraw it.** Ask them to upload it (SVG is ideal; PNG/JPG work too) rather than building a new mark from scratch. Embed it directly with a normal `<img src="their-logo.svg">` (or `.png`) in every slot the template's `{{MARK_SVG*}}` placeholders would otherwise fill — WeasyPrint renders both vector and raster files fine as long as the file sits next to the HTML (or you pass `base_url` pointing at its folder when calling `HTML().write_pdf()`). A few things to flag to the person when doing this: you can't automatically recolor their file to match different backgrounds the way a code-built mark can, so the logo-variation grid should only show combinations that actually work with the file(s) they gave you — if they only have one full-color version, don't fabricate a "reversed on dark" cell, just note in the misuse section that a white/reversed version doesn't exist yet and should be requested from whoever made the original. Use the uploaded logo's colors as the seed for the rest of the palette (extract the dominant colors by eye from what you can see in the file) rather than inventing an unrelated palette around it.

Don't promise legal certainty you don't have. Flag font licenses, AI-image provenance, and trademark considerations as things to check — don't assert they're cleared. See `references/accessibility_and_legal.md`.

## Two modes: create vs. codify — diagnose this first

Every request is one of two jobs, and confusing them produces the wrong book:

**Mode A — Create.** No identity exists; you're designing one. The full workflow below (intake → strategy → visual system → applications → export) applies as written.

**Mode B — Codify.** An identity already exists in the wild — a professionally built website, established materials, a real logo — but it's undocumented. The job is extraction and organization, not design. **In Mode B, inventing anything is a failure.** Signals: an established business, screenshots of a live site, language like "organize," "document," "for the staff," "so vendors stay consistent." What changes:

- **Strategy is archaeology.** Read positioning, differentiators, and proof points off their existing published materials. Every claim in the book must be traceable to something they already say publicly — the verified-accuracy standard below applies with full force.
- **Every fact is grounded or it doesn't ship — zero hallucinations.** Follow the evidence hierarchy in `references/web_grounding.md`: supplied master files > DevTools extractions > verbatim page text (pasted or live-fetched this session) > pixel samples > the person's statements. Model memory of the brand is class 0 — BANNED as a source, even for famous brands, even when it's probably right. No compositing remembered details onto sourced ones, and every date in the book (cover year, capture dates, next-review) follows the date discipline in that file: the machine's `date` output, the person's stated capture date, or a sourced historical fact — never a year typed from recall. Run the guided site capture (same file) during Step 1.5 so the excavation starts from canonical values.
- **Colors are sampled deterministically, then closed.** The exhaustive-palette rule (off-list colors are not brand colors) is mandatory content, but a palette that drifts between runs is worse than no palette — two runs against identical inputs MUST produce identical values. Discipline: (1) **master files outrank samples** — any color present in a supplied vector/logo master is recorded at its exact fill value and labeled canonical; screen-sampled values are recorded with a ±2/channel tolerance and marked provisional wherever the seeded provenance placement lives. (2) **Sample per element, not per pixel** — each palette entry derives from ≥5 pixel samples across ≥2 pages of the same element; record the mode; a single-sample entry doesn't ship. (3) **Surface census** — before locking the palette, enumerate every distinct brand-controlled surface appearing in the applications chapter (masthead, utility bars, footer, cards, product stages, banners) and verify each maps to a palette entry; a surface with no entry means the sampling missed a color. (4) **Exhaustive-palette claims require an exclusion list** — enumerate what was excluded and why (licensed/third-party art, machine chrome, photographic tones); no exclusion list, no exhaustiveness claim. (5) **Measurement units are single-sourced** — clearspace and minimum-size rules declare their basis once (e.g., "T = tile width") and every later rule reuses that token; width-vs-height ambiguity is a QA failure. (6) **The AI Brand Kit is canon** — palette, units, and minimums are written to the kit at sampling time and the book renders from those values, so book and kit cannot drift.
- **Provenance is firewalled from the body.** Where the evidence trail lives is a seed (`references/structural_seeds.md`), but the *register* is not: body pages state what the rule IS, never how it was learned. No "screenshots," "captures," "supplied," "sampled on [date]," "this engagement," or display-profile caveats outside the seeded provenance placement — a professional book asserts ("The navigation band is Surface Yellow") and lets the back-matter register prove it. The cover never names input materials; for unofficial codifications of third-party brands, the not-endorsed disclaimer appears exactly twice (a small cover/title status line and a full closing block), and the sampling caveats, asset dispositions, and open confirmations live only at the seeded placement. Run the confidence lint in `references/anti_fingerprint_checklist.md` before export.
- **If the brand hosts third-party IP, define the frame.** Trigger: excavation reveals licensed characters, partner logos, marketplace goods, or artist content rendered at full strength on the brand's own surfaces. When triggered, the book must distinguish the surfaces the brand controls invariantly from the zones where partner color/art lives at full strength, and state both invariants: brand surfaces never tint toward a guest, and a guest color's presence on-site confers no approval for its use in owned work. This is mandatory CONTENT; name it in the brand's own vocabulary or leave it unnamed (the concept table in `anti_fingerprint_checklist.md` retires this skill's house names for it).
- **Fonts: document behavior, don't guess names.** If the exact licensed webfont is unknown, describe the typographic signature (family class, weight, tracking), set the book in open-license equivalents, say you did so in an "In practice" note, and route font confirmation to whoever built the original identity.
- **Never redraw or fabricate assets.** Index the real logo files by filename with use-for guidance. If a variant doesn't exist (e.g., no reversed wordmark), say so in the book and route the request to the original designer — by name, with credit.
- **Governance and staff rules are first-class sections**, not footnotes: approved text pairings spelled out in prose, a shared-folder structure, "never screenshot the logo — pull from the folder."
- **Skip the visual-preference intake questions entirely** — the site already answered them.

Hybrid (real logo exists, nothing else does): run Mode A seeded by the actual logo file, per the existing-logo guidance above.

Two fully worked illustrative case studies — one per mode, with the decisions and revision patterns that recur in real delivery — are in `references/worked_examples.md`. Read it before Step 2 on any engagement that resembles either archetype. Two contrast-checked example palette/type systems are in `references/preset_systems.md`; offer a preset as a *seed* only when the brief maps cleanly onto its archetype, always mutate it, and never pass one off as bespoke.

**Verified accuracy standard.** Every specific factual claim in a brand book — awards, credentials, review counts, follower counts, years in practice, institutional affiliations — is treated as unverified until sourced from the client's own published materials or confirmed by the client directly. If it can't be sourced, it doesn't ship; ask rather than approximate. Proof points are the section where fabrication does the most reputational damage.

## What's in scope, and what's genuinely adjacent

A brand is bigger than any one document. Here's the full picture, so nothing gets quietly dropped and nothing gets promised that this skill doesn't actually deliver.

**Built by this skill, every time:** positioning, differentiation, voice/tone, messaging (tagline, elevator pitch, proof points), a real rendered mark plus variations, full color and type systems, imagery/pattern direction, accessibility-checked contrast, and — from Tier 2 up — UI components, a mini mockup, and real application examples.

**Built by this skill, if asked:** a longer-form origin/founder narrative (a few paragraphs beyond the elevator pitch — fold it into the Brand Essence section rather than adding a new page unless there's a lot of story to tell), a distinct mission/vision/values trio instead of the default single positioning statement (some people want the classic three-part format; it's a straightforward variant of Step 2), social bio copy and an email signature block (treat as additional application examples).

**Adjacent, not this skill's job — say so plainly rather than attempting it or silently skipping it:**
- **Naming** — this skill assumes a name exists. Naming a business is a different, earlier exercise; happy to help with that separately, but flag that it's a switch in task, not a brand-book subsection.
- **Trademark/domain/handle clearance** — flag it as something to check, and if web search is available in the session, a quick gut-check search (`"[name]" trademark`, domain availability) can surface an obvious conflict — but say clearly that this is a spot-check, not a clearance search, and a real one needs an attorney or a proper search service.
- **Actually building the website, social accounts, or print-production files** — this skill produces the brand book (the reference document), not the built artifacts. A mini mockup in the brand book shows what a site *could* look like; it isn't shippable code. Say this explicitly so nobody expects a live site to fall out of a PDF.
- **Packaging engineering, business-card die-lines, signage fabrication specs** — physical production specs (bleed, dielines, substrate) are a print/packaging designer's job; this skill can describe the *look* but not hand off production-ready files for physical goods.
- **A full content calendar or social strategy** — that's a separate, ongoing exercise, not a one-time document.
- **Pitch decks, one-pagers, or other standalone documents** — once the brand book exists, building one of these is a normal request for Claude's pptx/docx skills, using the brand book as the style reference; it's just not something this skill generates itself.

If a request clearly wants one of the adjacent items, do that as its own task using the finished brand book as the reference point, rather than trying to cram it into the brand book itself.

## Step 1: Figure out what you're working with

Check what the person has already given you — a prior conversation, an uploaded deck or old brand guide, a description of the business, reference images, a competitor they admire. Extract everything usable before asking for anything. People rarely arrive with a complete brief, and that's normal — your job is to fill the gaps efficiently, not to demand a complete dossier upfront.

**If the brief is reasonably rich** (you can already infer audience, tone, and category), state your read of it back to them in 2-3 sentences and move to Step 2. Don't interview someone who already told you the answers.

**If the brief is thin** (just "I need a brand for my [thing]" with little else), interview them using the `ask_user_input_v0` tool — the same tappable multiple-choice format used elsewhere in this conversation. This is much faster for the person than typing paragraphs, and it keeps you from guessing at things like tone or market position. Don't ask everything in one giant dump; run it in 2-4 short rounds of 1-3 questions each, using their answers to shape the next round. Full question bank, organized by theme (business basics, audience & positioning, voice, visual preference, scope/depth), is in `references/intake_questions.md` — pull from it rather than inventing questions from scratch, but adapt the wording to their actual business.

One thing worth asking early because it changes the whole shape of the deliverable: **how much do they want?** A solo creator's first brand book and an enterprise client's mature brand system are different documents. `references/asset_inventory.md` defines three depth tiers (minimum viable / strong launch / mature system) — ask which fits, or infer it from context (a "just getting started" founder almost never needs the mature tier).

## Step 1.5: The asset call — collect what exists before designing anything

Unless the person has clearly said they're starting completely from scratch, run one explicit asset request before strategy synthesis: a short plain chat message (not `ask_user_input_v0` — the tappable tool can't receive files) asking them to attach whatever already exists — logo files and every variant, symbols/motifs/patterns, screenshots of their site/social/storefront/packaging, photography they consider "them," fonts they own, any old guide or deck, and reference material from brands they admire. Then wait for the reply. The full request script, priority order, per-asset handling table, and the no-assets path are in `references/asset_call.md`. On Mode B engagements where the identity lives on a website, run the capture ladder in `references/web_grounding.md` — Claude works first, the user works last: rung 0 is Claude fetching the site itself (HTML, linked CSS for design tokens and @font-face names, logo SVGs, about-page copy) from just the URL; rung 1 is everyday screenshots plus pasted key copy with exact OS keystrokes; rung 2 is the one-click bookmarklet (`assets/brand_capture_bookmarklet.txt`); rung 3 (DevTools) is offered only to comfortable users or for one stubborn value. Ask only for named gaps the lower rungs left, and confirm the one-pass fidelity checklist before building. The reason this step is non-negotiable: the person is picturing their brand as the assets they already own, and a book built on their real materials matches that picture; a book that designs first and asks later competes with it. After assets arrive, echo a one-paragraph read of what you're taking from them ("your green is the anchor, the bee mark is untouchable...") before building — this is the cheapest moment to catch a mismatch, and it usually settles the Mode A/B question on its own. Everything received becomes canon: seed the palette from it, never redraw the mark, show only variants that actually exist, and log every supplied file in the asset manifest.

## Step 2: Synthesize the strategy

Before touching colors or fonts, write the strategic core:

- **Positioning statement** — who it's for, what category it competes in, what it promises, why that's credible coming from this brand specifically
- **Differentiation** — what makes it not interchangeable with the obvious alternative; if you can't articulate this, say so honestly rather than papering over it with adjectives
- **Voice and tone** — 3-5 adjectives, each with a "sounds like / doesn't sound like" pair so they're actionable rather than vibes (e.g. "warm, not saccharine")
- **Core message hierarchy** — one tagline-length statement, one elevator-pitch paragraph, 3-4 supporting proof points

Show this to the person before building the full visual system around it. It's much cheaper to fix the strategy in a paragraph than to discover three sections of imagery direction were built on a positioning nobody actually agreed with. A quick "here's the strategic core — does this sound like you, or off?" is worth the pause.

## Step 2.5: Choose the document's posture (never skipped, even for rich briefs)

Before any layout work, decide what KIND of document this brand book is — this is the single biggest structural decision and it is never inherited from a template. Derive a recommended posture from the strategy using `references/document_postures.md` (strategy-led narrative, voice-led guide, color/visual-first, spec/governance book, one-sheet system of record, or rebrand-implementation package), then present it to the person via `ask_user_input_v0` with a one-line "because" and 2-3 genuinely different alternatives (see `references/intake_questions.md` Round 2.5). Strategy proposes, the person disposes. The confirmed posture sets the section spine, depth emphasis, heading register, and completeness floor (the floor matrix in `document_postures.md` scales required coverage to the posture and brand type — form is free, floors are not). This applies in BOTH modes: Mode B codify books very often land as spec/governance or one-sheet postures because their primary readers are staff and vendors, but that's a derivation to confirm, not an assumption. Professional brand books differ from each other at this level first: a voice-led quick guide, a color-first system, a versioned rebrand package, and a dense one-sheet are different documents, not one document restyled — and this skill's output must have that same range.

## Step 2.6: Derive the central question (after posture, before any outline)

State, in ONE sentence, what this specific book is protecting the brand from — the failure mode most likely when other people produce material in its name — then validate it against the Step 1.5 evidence and confirm it with the person inside the Step 2 strategy check-in. The confirmed question governs page allocation (its territory gets the deep chapters; everything else compresses to its floor), the opening move (books may open ON the problem, not on generic essence), and which rules get named versus silently enforced. If no question survives evidence-validation, fall back to the posture's default spine — a legitimate outcome. Full procedure, examples, and guardrails in `references/central_question.md`. Record the question as an HTML comment in the book and a field in the AI Brand Kit. This is the step that makes two books structurally different for a REASON: professional books never share a skeleton because they were never answering the same question.

## Step 2.7: Synthesize the author (after the question, before any outline)

Compose the author of THIS book — a one-paragraph profile (handle, design lineage, era of formation, two convictions, one blind spot, one pet move, prose-cadence target bands) per `references/author_synthesis.md`, plausible for this brand's category and compatible with the evidence and central question. Every downstream judgment call — seeds, register, page rhythm, emphasis, sentence cadence — is made AS that author. The profile must pass the non-resemblance diff against every prior profile in `references/structural_digests.md` (lineage never repeats consecutively; convictions, pet moves, and blind spots never recur; composite matches fail). Spend the imperfection budget: 1-2 authored idiosyncrasies traceable to the profile, never touching floor content. For choices that remain genuinely free after question + seeds + profile, commit them with `python3 scripts/committed_choice.py "<brand>" "<date>" "<label>" "optA|optB|..."` rather than letting habit pick — log each in the SEEDS comment. This is the positive divergence engine: the professional corpus varies because it had independent authors, and this step is how each engagement gets one.

## Step 3: Build the visual system

With strategy locked, build:

**Color** — Choose a primary, secondary, and neutral palette that's traceable back to the strategy and tone adjectives — but "traceable to strategy" doesn't mean "muted and tasteful" by default. A confident, loud brand can and often should have a saturated, high-contrast, even slightly uncomfortable palette; a brand explicitly trying to stand out from a polite category is poorly served by the same soft neutral-plus-one-accent formula every other brand book reaches for. Let the tone adjectives actually drive saturation and contrast, not just hue. If they uploaded a logo or reference images, ground the palette extraction in those first. The one non-negotiable constraint is accessibility, not restraint — always check primary text/background pairings against WCAG contrast minimums (4.5:1 normal text, 3:1 large text and UI elements) with `scripts/check_contrast.py`, and if a bold color fails as text, keep it for backgrounds/large surfaces/accents and pair it with a text-safe variant rather than abandoning boldness altogether (see the Hearth & Ember example in this skill's own history: a vivid rust stayed vivid everywhere except body text, where a darker variant took over). Provide hex and RGB at minimum; add CMYK/Pantone only if they mention print production needs.

**Typography** — Don't default to "editorial serif + clean sans" out of habit — that's one vibe among several, and it's been overused as the safe choice. Look at the actual tone adjectives from Step 2 and pick a display face that matches how loud, warm, sharp, or playful this brand actually is; `references/font_pairings.md` organizes real options by vibe (bold/blocky, geometric/modern, editorial, playful, maximalist) rather than defaulting to the same pairing every time. Run `python3 scripts/setup_fonts.py <display-slug> <body-slug>` before touching the template — this fetches the actual font files via npm's `@fontsource/*` packages (which bundle real Google Fonts files) and prints an `@font-face` block to paste in. This matters because this environment can't reach `fonts.googleapis.com` directly, so just naming a font without embedding it will silently render as a fallback system serif/sans regardless of which one you picked — the fetch-and-embed step is what makes the distinctive choice actually show up in the PDF instead of defaulting back to something generic. Default to license-safe families (which every `@fontsource` package already is) unless the person specifies they have other licensed fonts. Define a basic type scale (display, h1-h3, body, caption) usable across web, print, and slides.

**Imagery and graphic language** — Describe photography style, illustration approach, iconography, and any recurring motif or pattern in concrete enough terms that someone shopping stock photos or briefing a photographer would know what to pick. Tie it back to the tone adjectives.

**Logo system (a real rendered mark, not just a spec)** — Build an actual small geometric mark in inline SVG using `references/mark_techniques.md` (a monogram badge or a simple 2-4 shape motif) — this is the single biggest difference between a brand book that feels finished and one that feels like a placeholder document, and it's achievable with basic shapes even though full illustrative artwork isn't. Produce it at the four sizes the template needs, show it in a logo-variation grid against 4-5 backgrounds, and document clear space, minimum size, and explicit misuse examples ("don't add glow effects," "don't recolor outside the palette," "don't place on busy backgrounds").

## Step 4: Map applications

Show the system in use, not just listed in isolation. At minimum: a social post template, a presentation title slide, a document header/footer, and a web hero treatment, described or laid out well enough to be copied. Pull realistic examples from `references/pdf_brand_book_structure.md`'s archetype notes (startup tech / boutique retail / nonprofit / personal-professional) if useful as a starting point, but write actual content for *this* brand, not generic placeholder copy.

## Step 5: Assemble and export the PDF

Treat `assets/brand_book_template.html` as a **component quarry, not a document**: reuse its CSS plumbing and proven page patterns, but write a bespoke `<body>` assembly for every book — section set, order, and page weights from the confirmed posture (`references/document_postures.md`) as governed by the central question (Step 2.6), one deliberate choice per variation axis in `references/layout_grammar.md` (cover archetype, contents treatment, heading register, divider strategy, voice-section format, do/don't format, specimen format, mark-chapter format, applications format, closer), and the seven seeded structural choices from `references/structural_seeds.md` (rule-tier vocabulary, voice-trait count/format, rewrite format, provenance placement, document-control furniture, doctrine-naming, contrast-evidence format) — each derived from brand attributes and declared with justifications in a `<!-- SEEDS -->` comment, with component counts varied on purpose. Never ship the quarry's assembly order, divider positions, or furniture copy as-is; every `{{*_HEADING}}` is written fresh in the brand's chosen register. Draw sections from the inventory in `references/pdf_brand_book_structure.md` and verify the posture's completeness floor is met. Restyle proportions (margins, heading scale, column rhythm) to the brand rather than inheriting the quarry's editorial defaults — the rendering pipeline is fixed; the design is not. Populate every section with this brand's actual content — never ship template placeholder text. Keep each `{{HEX_*}}` field to a single hex value — if a color needs a separate text-safe variant (see Step 3), put that nuance in the color usage notes paragraph instead, since cramming two hex codes into one swatch label wraps awkwardly. For Tier 3 books, add extra pages (brand architecture, governance, expanded asset index) by duplicating the `<div class="page">...</div>` block pattern at the end of the body — the CSS classes (`.section-label`, `h2`, `h3`, `p`, `ul`) all carry over, no new styles needed.

Convert to PDF with **WeasyPrint** — it's not preinstalled, so run `pip install weasyprint --break-system-packages` once per session before first use. WeasyPrint respects CSS physical units and `@page` sizing exactly, which matters for full-bleed colored pages like the cover. (The alternative, `pdfkit`/`wkhtmltopdf`, is preinstalled but most environments ship an unpatched-Qt build that silently shrinks full-bleed layouts by a fixed, non-configurable ratio — avoid it for this template unless WeasyPrint genuinely can't be installed.)

```python
from weasyprint import HTML
HTML('brand_book.html').write_pdf('brand_book.pdf')
```

The template already declares `@page { size: 8.5in 11in; margin: 0; }`, so no extra sizing options are needed. If you ran `setup_fonts.py`, the `fonts/` folder it created needs to sit next to `brand_book.html` when you render — if they're not in the same directory, pass `base_url` pointing at the HTML's folder: `HTML('brand_book.html', base_url='.').write_pdf('brand_book.pdf')`. Same applies if you're embedding an uploaded logo file per Step 3.

Render the result to PNG (`pdf2image`'s `convert_from_path`) and visually check at least the cover, a text-heavy page, and the color swatch page before calling it done — copyfitting issues (overflowing text, awkward page breaks) are the most common failure mode of HTML-to-PDF generation and are easy to catch this way but easy to miss by just trusting the code ran without error.

**Then run `references/anti_fingerprint_checklist.md` in full before export** — on Mode B books this now includes the confidence lint (no excavation lexicon on body pages or the cover), the palette-determinism gate (master-outranks-sample, surface census, exclusion list, single-sourced units, kit-as-canon), and the grounding & date gate (every fact evidence-classed per `references/web_grounding.md`, zero model-memory claims, machine-dated cover and control blocks, year grep accounted) — plus the retired-copy grep (now including the coinages caught in the LEGO/Rolex sibling pair), the concept-level ban check (mandatory rules expressed in the brand's own vocabulary or left unnamed, never in house coinages), the structural gates (posture confirmed, central question governs the outline, seeds declared and followed, cover/dividers deliberately chosen, counts varied, completeness floor met AS OUTCOMES, escalation owner named), the mechanical digest diff against every prior output in `references/structural_digests.md` (now including author-profile non-resemblance, idiosyncrasy reuse, and the prose-cadence rule — run `scripts/prose_fingerprint.py` on the final HTML and verify the book hits its own author's bands while breaking from the house hand), and the sibling test: if this book's skeleton or vocabulary would sort into a pile with the skill's previous outputs, it fails and gets re-expressed before delivery. After delivery, append the new book's digest to `structural_digests.md` and remind the person to commit the updated corpus to their repo copy. Save the final file to `/mnt/user-data/outputs/` and use `present_files`.

Also generate a short asset manifest (filename, source, license, last-updated) per `references/accessibility_and_legal.md` — even a few lines of provenance tracking saves real pain later, especially for fonts and any AI-assisted imagery.

## Step 5.5: Generate the AI Brand Kit (companion file)

Alongside the PDF, always generate `[brand-slug]-BRAND.md` from `assets/brand_kit_template.md` — a machine-readable companion the person can hand to Claude (or any AI) so future generated content stays on-brand without re-explaining anything. The PDF is for humans; this file is the same system compiled into *instructions for an AI*.

Rules for writing it well:
- **Commands, not descriptions.** "Never introduce colors outside this list," not "our palette reflects restraint." Every section should be enforceable.
- **Self-contained.** Assume the reading AI has no other context: closed palette with hex values and per-color jobs, approved text pairings, type rules with actual tracking numbers, voice adjectives with say/not-this pairs, verbatim identity facts and the ONLY approved proof points, and required compliance language.
- **The mark travels as code.** Embed the final mark as inline SVG so a future AI can place it exactly rather than redrawing it. Mode B: reference the client's real logo files by filename instead, restate the don't-fabricate-variants rule, and include a live-text fallback recipe for sessions where the files aren't attached.
- **Steer graphics to code, not image generation.** Include the instruction that brand graphics must be built as HTML/SVG with exact hex values and rendered — text-to-image models can't reproduce exact colors or logos, and this single instruction is the difference between passable and unquestionably on-brand.
- **Per-format recipes.** Write concrete recipes for the formats this brand will actually use (social post, story, email, print), each specifying surface color, headline treatment, body limits, mark placement, and CTA style — including one worked promotional example so discount/sale asks land in-voice rather than in urgency-spam.
- **End with a pre-flight checklist** the reading AI must run before presenting output (colors, pairings, mark, voice, claims, compliance, type).
- Keep it under ~400 lines so it fits in any AI's context alongside an actual task.

When delivering, tell the person what it's for in one or two sentences: save it next to the PDF, and attach it whenever asking any AI for brand content — in Claude, adding it to a Project's knowledge means never attaching it manually again.

## Step 6: Offer revision, not just delivery

Brand work is iterative almost by definition. After delivering the PDF, explicitly invite section-level feedback ("anything in the color system, voice, or applications you want to push on?") rather than treating the first draft as final. If they ask for a change, edit the HTML template in place and re-export rather than rebuilding from scratch — keep locked sections untouched.
